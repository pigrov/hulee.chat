# Inbox V2 Direct Messenger Matrix

Status: `active baseline`  
Owner task: `INB2-ARCH-008`  
Last verified: `2026-07-10`  
Scope: Telegram, WhatsApp and MAX personal-session direct accounts only.

## Outcome

Hulee currently has a tested account/auth/session/probe foundation for all three
providers, but it does not yet have a composed direct-account message listener
or dispatcher for any of them. Consequently:

- exact auth/session/probe rows can be `present` with Hulee automated evidence;
- session health and SourceAccount synchronization are present only for their
  narrow runtime scope;
- every private/group message, roster, multi-account, history, echo, media,
  lifecycle, receipt and reaction row starts as `absent` with missing Hulee
  automated/live evidence;
- no RIK `OK`, `PARTIAL`, smoke result or production revision is a Hulee status;
- the current shared `directAccountCapabilities` object and admin catalog
  overstate direct messaging support and are recorded as gaps, not evidence.

Completing this document completes the architecture baseline only. Epic 8 and
the `G2`, `G4` and `G7` gates remain open until their required matrix cells pass.

## Canonical Grain

The canonical key is `(surfaceId, capabilityId)`. The normalized, machine-readable
ledger is `docs/product/inbox-v2-direct-messenger-cells.csv`; it owns every
individual cell, evidence requirement, evidence link, limitation and full
backlog-task ID. The wide tables below are a human-readable projection and must
not be used to transfer one provider's evidence to another provider.

The future `INB2-DMX-001` contract fixtures should use the same surface and
capability IDs and generate or validate this projection so two truth sources do
not drift.

The ledger currently contains `504` unique cells across the six direct-message
surfaces, three account surfaces and one Hulee platform surface. Pipe-delimited
IDs inside a CSV field are sets, not status roll-ups.

## Status And Evidence Vocabulary

Each surface cell is rendered as `providerSupport/huleeImplementation`.

| Field                   | Allowed values                                               | Meaning                                                                                       |
| ----------------------- | ------------------------------------------------------------ | --------------------------------------------------------------------------------------------- |
| `providerSupport`       | `supported`, `unsupported`, `unknown`, `not_applicable`      | Static support for the exact provider surface and atomic capability.                          |
| `huleeImplementation`   | `present`, `absent`, `limitation_enforced`, `not_applicable` | Current Hulee implementation; it is deliberately independent from backlog task state.         |
| `providerEvidence`      | `PROV-*`, `missing` or `not_required(reason)`                | Exact API/transport behavior, version, object kind, date and scope.                           |
| `evidenceRequirement`   | `auto` or `auto+live`                                        | Minimum evidence needed for this exact cell; stored in the normalized ledger.                 |
| `automatedEvidence`     | `AUTO-*`, `missing` or `not_required(reason)`                | Hulee-owned test for the exact cell, command/result/date and working-tree or commit identity. |
| `liveEvidence`          | `LIVE-*`, `missing`, or `not_required(reason)`               | Hulee live provider smoke for the exact surface/object kind.                                  |
| `limitationOrCondition` | `LIM-*` or an explicit condition                             | Provider semantics, adapter gap, Hulee gap, product scope or still-unknown behavior.          |

`partial` is not a status. Split direction/content/operation into atomic rows and
put conditions in the limitation register. Runtime states such as rate limited,
expired or temporarily unavailable belong to a binding/operation, not this
static capability baseline.

### Gate Rules

- A required `supported` cell passes only when Hulee is `present`, exact
  automated evidence exists and required live evidence exists.
- `unsupported` or `not_applicable` passes only through an accepted limitation
  and a fail-closed negative test when the operation is visible in UI/API.
- `unknown` never passes a release gate.
- Code without exact evidence does not pass.
- RIK evidence, a transport library or another product cannot populate Hulee
  `AUTO-*` or `LIVE-*` evidence.
- There is no majority roll-up: every required cell must pass.
- Live smoke must be repeated after a provider/transport version change and
  before the release gate that consumes it.
