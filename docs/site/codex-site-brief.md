# Hulee Site Brief For Codex

Перед каждой новой итерацией по маркетинговому сайту сначала перечитать этот файл и исходную спецификацию `docs/site/Hulee_структура_и_контент_лендинга.docx`.

## Текущее состояние

- Активный `apps/site` сброшен до заглушки с логотипом Hulee.
- Старый дизайн и связанные ассеты упакованы в `docs/site/archive/hulee-site-current-design-20260716-213030.zip`.
- Не удалять брендовый слой: `apps/site/public/brand`, favicon-файлы в `apps/site/public`, `apps/site/public/icons`, `apps/site/public/apple-touch-icon.png`.

## Главная цель сайта

Сайт должен быстро объяснять, что Hulee - это рабочее пространство для клиентских и внутренних обращений: каналы, команда, история, файлы и контроль данных в одном продукте. Главный рыночный акцент - оплата не за каналы и не за учетные записи операторов, а за реальный ресурс хранения данных, истории и файлов.

## Визуальные правила

- Один доминирующий visual на секцию.
- В первую очередь показывать реальный интерфейс и реальные продуктовые сценарии.
- Не делать галерею однотипных 3D-иллюстраций.
- Не использовать фейковые клиенты, отзывы, фотографии людей и недоказуемые метрики.
- Текст должен быть простым русским языком, без внутренней технической терминологии в главной подаче.
- Light/dark темы проектировать сразу, но не плодить отдельную логику без нужды.
- Mobile layout - отдельный продуманный вид, а не сжатая desktop-таблица.

## Дизайн-токены

Базовая сетка: max width 1240-1280 px, 12 колонок.
Секции: desktop gap 140-180 px, mobile gap 88-112 px.
Типографика: H1 56-72 desktop / 38-48 mobile, section H2 44-56, body 18-21, line-height 1.5-1.65.
Радиусы: cards 20-24 px, inputs/buttons 12-16 px.

Light: background `#FDFDFF`, text `#11162B`, surface `#FFFFFF`, border `#DFE4F2`, accent `#6D5DFB`, blue `#3185FF`, green `#27C98B`.

Dark: background `#11162B`, surface `#171D35`, border `#2A3354`, text `#F6F7FF`, muted `#AEB8D5`, accent `#7C6CFF`, green `#38D39F`.

## Термины для публичного текста

- `inbox` -> `единое окно` или `рабочее пространство`.
- `storage-based` -> `модель оплаты за объем хранения`.
- `retention` -> `срок хранения`.
- `tenant boundary` -> `данные компании хранятся отдельно`.
- `on-prem` -> `размещение на серверах компании`.
- `provider-costs` -> `внешние расходы поставщиков`.
- `lead-response` -> `обработка заявок`.
- `enterprise` -> `крупные компании` или `корпоративный вариант`.
- `core` -> `единая версия продукта`.
- `white-label` -> `оформление под бренд компании`.

## Правила реализации

- Не хардкодить брендовые ассеты в компонентах: брать `defaultBrandProfile`.
- Видимые строки держать в i18n-словарях.
- Цвета в компонентах не хардкодить; использовать CSS variables/design tokens.
- Каждый новый блок делать как самостоятельную итерацию: добавить контент, стили, assets, проверить desktop/mobile и light/dark.
- Не заявлять интеграции как готовые, если они не доступны в продукте.
- Калькулятор должен разделять Hulee-стоимость и внешние расходы поставщиков.

## QA Перед Следующей Итерацией

- Первый экран отвечает: что это, для кого, в чем модель оплаты.
- Нет фейковых клиентов, отзывов, фотографий и метрик.
- `0 руб. за канал/оператора` не повторяется навязчиво в каждой секции.
- Контраст light/dark читаемый.
- Кнопки ведут в реальные или явно согласованные сценарии.
- Скриншоты и интерфейсные изображения актуальны для текущего направления дизайна.
